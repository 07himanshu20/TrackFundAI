1. What this document is

TrackFundAI is a multi-tenant SaaS for Indian Alternative Investment Funds (AIFs). The General Partner uploads a fund Excel file — any layout, any naming convention — and the system turns it into a structured dashboard (Portfolio, Fund Performance, Fund Reporting, LP Management, Compliance).

The hard problem: every fund's Excel looks different. Different sheet names, different column headers, different ordering, different abstractions, different units, sometimes different languages. We never hardcode “if sheet name is X do Y”. Instead, every decision about “what is this row/column/cell?” is made by Google Gemini against a fixed canonical schema, and every number you see is computed by Python using only catalogue-validated inputs (never trusting Gemini's claimed numeric values).

The pipeline is organised as a sequence of numbered Passes. Each Pass has one job. Each Pass is Gemini-driven where semantic judgement is needed, and Python-only where arithmetic is needed.

2. Table 1 — All Passes (what each Pass does)

A “Pass” is a discrete stage in the import pipeline that runs once per file (or once per sheet/scheme inside a file). Decimal Passes (1.5, 2.5, 2.6, 3.5, 6.5, 6.6) are refinements that were inserted between the original integer Passes as the system matured.

Pass Name Role — what it does, why it exists
Pass 1 Sheet Domain Classification One Gemini call sees a preview of every sheet in the workbook and classifies each one into one of ~19 fixed domains (e.g. portfolio_companies, capital_calls, distributions, nav_records, waterfall_carry, kpi_tracker, scheme_lifecycle, unknown). Result is a sheet → domain map that decides which importer will read that sheet. Cover/Summary/Index sheets land in unknown and are never used as data sources.
Pass 1.5 Section Sub-Domain Classification Many sheets contain multiple stacked sections (e.g. a single Compliance sheet may hold both a per-company tracker and a fund-level filings table). Pass 1.5 sends every banner/section-title row to Gemini in one batched call and assigns each section to a sub-domain. This is how a single Excel sheet can populate two different Django models cleanly.
Pass 2 Column-Header to Canonical-Field Mapping For each classified sheet, Gemini maps the raw Excel column headers (“Cmp Nm”, “Sec”, “Cmt INR Cr”…) to canonical field names (company_name, sector, commitment_amount…) declared in canonical_schema.py. Runs in parallel across sheets via ThreadPoolExecutor with a 2-layer retry. Output: per-sheet raw_header → canonical_field alias map.
Pass 2.5 Per-Sheet Layout Detection One Gemini call per sheet returns the sheet's internal anatomy: which rows are noise (banner / disclaimer / footnote), which row is the real header, where data starts and ends, where sub-tables live, and which columns are formula-derived in Excel. Replaces brittle Python heuristics that broke on sheets whose first rows are not headers.
Pass 2.6 Column Semantic Role Classifier For each horizontal section Pass 2.5 found, classifies every column into a semantic role: per_period_amount, cumulative_total, ratio_percent, identifier, metadata_text, derived_indicator, unknown. Pass 3.5 uses these roles to refuse to extract a per-period value from a cumulative column (which was the root cause of an early Preferred-Return bug).
Pass 3 Value Classification (Label → Canonical Category) Batch Gemini classifier that maps free-text labels into canonical categories (kpi_types, pl_line_items, investor_types, nav_components, waterfall_components, etc.). Replaces ~1100 hardcoded English keywords that previously broke on non-Indian English files.
Pass 3.5 Universal Explicit-Value Extractor Scans every sheet for cells whose labels match canonical fund-performance metrics (nav, moic, tvpi, net_irr, total_called_capital, etc.). When multiple cells claim to be the same metric, Gemini disambiguates (e.g. picks gross vs net based on the variant tag). Writes DerivedMetric rows with formula_expression='(direct value imported)' so Pass 4 sees them as authoritative and won't re-derive.
Pass 4 Fund-Metric Derivation (Gemini Ranked Formulas) For each derivable metric not already extracted, asks Gemini for a ranked list of candidate formulas. Python AST-validates that every variable in each formula exists in the actual catalogue (so Gemini can't hallucinate variable names). Walks candidates top-down, evaluates the first viable one using only catalogue values, persists with full provenance. Gemini never supplies numbers — only formulas.
Pass 5 Extraction Completeness Audit Diagnostic-only. Compares the number of rows actually imported per sheet against the count Pass 1 said the sheet contained. Surfaces silent failures (e.g. Pass 2 returning low-confidence mappings for an entire sheet). Does not modify data.
Pass 6 Per-Row Metric Completer Walks every fund-data Django model (Investment, PortfolioCompany, …), finds columns that are null on most rows (e.g. Investment.irr_pct), and asks Gemini ONCE per model for a per-row formula using the other available row fields. Evaluates the formula per row in Python. Result: per-company IRR / ownership % / holding period get filled when derivable. Zero hardcoded field lists.
Pass 6.5 Universal KPI Projection Walks the Pass 2 column-mapping for every sheet; whenever a column is mapped to a canonical KPI field (revenue, ebitda, headcount, mrr, …), projects those period values into PortfolioKPI rows. This is how the KPI matrix on the Fund Performance page gets populated from any sheet that has KPI columns.
Pass 6.6 KPI Percentage Derivation After 6.5 fills raw amounts, derives missing percentage KPIs (EBITDA %, gross margin %, growth %) using the same ranked-candidate-formula architecture as Pass 4. Solves the case where Excel has “EBITDA” but not “EBITDA %”.
Pass 7 CarriedInterest Writer Pure writer. Reads carry / waterfall DerivedMetric rows produced by Passes 3.5, 4, 8 and persists the CarriedInterest model that the Fund Reporting page reads. Also runs a mathematical identity check (Return of Capital + Preferred Return + GP Catch-up + Carry Base ≈ Total Proceeds) and logs the variance so audit can surface bad component picks.
Pass 8 Direct Waterfall Computation For waterfall sheets (European or American model) we send the entire sheet content as a labelled R<row> C<col>="<value>" grid, plus LPA terms and capital flows, in ONE Gemini call. Gemini returns the 4 primary metrics (carry_base, carry_amount_gross, gp_clawback_provision, carry_amount_net) with source-cell citations and reasoning. Python sanity-checks net == gross − clawback and backfills if Gemini contradicts itself. Born from the observation that one prompt with the full sheet outperformed the layered pipeline.
Two architectural rules that apply to every Pass:
  • No hardcoded keywords / sheet names / column names — every semantic decision is Gemini's.
  • Gemini never generates numbers that reach the database. Gemini proposes formulas; Python evaluates them using only values present in the catalogue. Any formula referencing an unknown variable is rejected.

Frontend ↔ backend communication pattern

The frontend is a vanilla-JS SPA — frontend/index.html is the shell, and every page is a <div class="v5-page"> swapped in by JS. All backend calls go through Auth.apiGet(url) / Auth.apiPost(url, body) in auth.js, which attaches the JWT and handles refresh.

  • Read pattern: page mounts → Auth.apiGet('/api/<app>/<resource>/?fund=<id>') → render rows.
  • Write pattern: the only writes from the SPA are file uploads to /api/dataimport/jobs/. Everything else (KPIs, NAV, carry) is computed by the import pipeline.
  • Provenance pattern: click any KPI tile → the side panel reads /api/dataimport/derived-metrics/?fund=<id> and shows formula, reasoning, source cell, confidence for that metric.

7. Semantic analysis — sheets, sections, rows, columns, fields

“Semantic analysis” in TrackFundAI means: understanding what something MEANS, not what it LOOKS like. The same data can appear in a thousand layouts. We solve this with Gemini at each level of granularity:

GranularityThe question we ask GeminiResult
Sheet “Given the first ~40 rows and the sheet name, which of these 19 domains is this sheet?” One of: portfolio_companies, capital_calls, distributions, nav_records, waterfall_carry, kpi_tracker, scheme_lifecycle, investor_register, fund_pl, fund_balance_sheet, fund_cashflow, compliance, governance, exits, fees_expenses, commitment_summary, bva, scheme_terms, unknown.
Section (banner row inside a sheet) “Inside this multi-section sheet, what is the sub-domain of each banner?” Sub-domain per section, e.g. one Compliance sheet may have section_A=per_company_filings and section_B=fund_filings.
Layout (header row, data range) “Which rows are noise, which row is the real header, where does data start/end, are there multiple sub-tables?” Layout map per sheet. Used as the PRIMARY row reader — replaces the older Python heuristic that broke on disclaimer/banner rows.
Column header “Which canonical field is ‘Cmt INR Cr’?” Canonical field name from canonical_schema.py (commitment_amount). Built per sheet, cached.
Column role (semantic) “Is this column a per-period value, a cumulative total, a ratio, an identifier, or metadata?” Semantic role per column. Pass 3.5 uses this to refuse mis-extractions like “don't extract a per-period amount from a cumulative column”.
Row “Is this row a real entity, a subtotal/total, a repeated header, a serial-number row, or a footnote?” real_entity rows are written; everything else is junk and skipped.
Cell value / label “Which canonical category does the label ‘Cost of Sales’ / ‘COGS’ / ‘Direct Cost’ map to?” Canonical key (cogs). Replaces ~1100 hardcoded keyword aliases that broke on non-Indian English.
Metric (fund-level) “Among these 3 cells all labelled ‘NAV’, which is the authoritative one (gross or net)?” Authoritative cell index + variant tag, written into DerivedMetric with full provenance.

8. Handling different Excel formats (names, ordering, abstraction)

Different sheet names

We never key off sheet names. Pass 1 sees the content of every sheet and decides its domain. A sheet named “Tab 5”, “Cap Calls”, or “Drawdowns FY24” can all be classified as capital_calls. A sheet named “Capital Calls” that actually contains a summary table will be re-classified by what its content looks like, not by the name.

Different column names

Pass 2 reads the actual column headers + a few sample data rows and emits a mapping into canonical fields. “Cmp Nm”, “Investee Co”, “Portfolio Company” all resolve to company_name. The Python code reads values via _find_col_str(row, 'Company Name', 'Company', 'Name') which uses a 5-pass fuzzy matcher (exact → case-insensitive → suffix → loose substring with guards) on top of Gemini's alias map.

Different ordering of sheets / sections

Order doesn't matter. The orchestrator iterates the sheet → domain map and dispatches each sheet to its domain-specific importer regardless of position in the workbook. Within a sheet, Pass 1.5 finds every section banner so a fund-level summary above per-company rows (or vice-versa) is handled identically.

Different architectures (single-sheet vs multi-sheet vs nested)

  • One sheet per domain: classical case. Each sheet is independently imported.
  • Multiple sheets contributing to one domain: e.g. one Excel may have separate sheets for Cat-I, Cat-II, Cat-III investors. All three classify as investor_register; importer iterates all of them and de-duplicates by (organization, pan).
  • One sheet hosting multiple domains: Pass 1.5 splits it into sub-domains and each section is dispatched independently.
  • Vertical (key-value) layout: e.g. Scheme Terms sheets that read "Hurdle Rate: 8%". extract_structured_metadata handles these without confusing them for tabular sheets.

Different abstractions (gross vs net, fund vs scheme, ₹ Crore vs $ Million)

  • Metrics that have multiple variants (gross vs net) are tagged at extraction via classify_metric_variant. The Variant Tag travels with the DerivedMetric, and downstream readers (carry engine, dashboard) honour a variant_default declared in the canonical schema.
  • Currency + numeric unit are detected per sheet via detect_currency_and_unit: returns {currency: 'INR', unit_multiplier: 10000000, unit_label: 'Cr'}. Values are stored normalised so “₹120 Cr” and “1.2 bn” both land as the same canonical number.
  • Fund-level vs Scheme-level: every model has a scheme FK; fund-level aggregates are computed on read by summing across the fund's schemes.

Different arbitration (multiple cells claim to be the same metric)

When Pass 3.5 finds N cells all labelled like “NAV”:

  1. If all N cells hold the same value → pick the first deterministically (no Gemini call).
  2. If they disagree → call select_authoritative_source(metric, candidates); Gemini returns a chosen_index with reasoning.
  3. If Gemini declines to choose → the metric is left for Pass 4 to derive from first principles instead of guessing.

9. The canonical schema — the single source of truth

backend/dataimport/canonical_schema.py is the dictionary of every domain, field, value category, enum, and metric the system can ever talk about. Nothing else in the code hardcodes a name — every other file references this catalogue.

What the canonical schema defines

  • SHEET_DOMAINS — the 19 domains a sheet can be classified as (Pass 1's allowed outputs).
  • DOMAIN_FIELDS — for each domain, the canonical fields (with description, type, units) that Pass 2 is allowed to map to.
  • CANONICAL_VALUE_CATEGORIES — for category-based classification: e.g. pl_line_items declares revenue, cogs, ebitda, … with descriptions. Used by Pass 3 to map a label to a category.
  • CANONICAL_ENUM_TYPES — closed enum sets: exit_type, distribution_type, valuation_methodology, quoted_status, …
  • CANONICAL_METADATA_FIELDS — for vertical key-value layouts: scheme_lifecycle (tenure, hurdle, carry, fee …) and fund_identity (SEBI reg, PAN, GSTIN, GIFT City flag …).
  • DERIVABLE_FUND_METRICS — for Pass 4: per metric, its label, description, unit, variant_default, and which inputs Gemini is allowed to assume.
  • VALUE_TYPE_TO_ALLOWED_COLUMN_ROLES — the compatibility matrix Pass 3.5 uses to refuse extracting (say) a per-period amount from a cumulative-total column.

How data is mapped to the canonical schema

Raw Excel Gemini Pass that decides Canonical schema entry Storage ───────── ───────────────────────── ─────────────────────── ──────── sheet ──────► Pass 1 (classify_sheets) ──────► SHEET_DOMAINS key ──────► (in-memory route) section ─────► Pass 1.5 (classify_sections)────► SHEET_DOMAINS sub-key ──────► (in-memory route) header ─────► Pass 2 (map_columns_for_sheet)──► DOMAIN_FIELDS[field] ──────► ImportFile.column_mapping label ─────► Pass 3 (classify_labels) ──────► CANONICAL_VALUE_CAT… ──────► PortfolioKPI, P&L row enum ─────► Pass 3 (classify_enum_values)──► CANONICAL_ENUM_TYPES ──────► ExitEvent.exit_type, etc. metric ─────► Pass 3.5 + Pass 4 + Pass 8 ────► DERIVABLE_FUND_METRICS ──────► DerivedMetric (provenance)

The canonical schema is the “contract” every Pass is held to: Gemini's outputs are validated against it. If Gemini returns a domain that's not in SHEET_DOMAINS or a field that's not in DOMAIN_FIELDS, the orchestrator rejects the response and either retries or falls through.

10. How we get to 100% accuracy

The accuracy properties are enforced by architectural rules, not by tests alone:

  1. Gemini judges, Python computes. Gemini is allowed to choose which formula applies and which cell is authoritative. Python is the only thing that performs arithmetic, and only on values that exist in the catalogue.
  2. Hallucination guard (AST validator). Every formula Gemini returns is parsed into an Abstract Syntax Tree. We extract every variable name. If any variable is not in the actual catalogue, the formula is rejected and we try the next ranked candidate.
  3. Idempotency. All writes use update_or_create keyed by natural identifiers (e.g. (organization, pan) for Investor, (scheme, metric_key, variant) for DerivedMetric). Re-importing the same file produces the same database state — no duplicates, no drift.
  4. No phantom rows. Every importer calls _is_junk_row(name) before writing. Subtotal / total / header / serial / footnote rows are filtered out by Pass 3 classification (not by hardcoded keywords).
  5. No missing rows. Pass 2.5's layout map gives explicit data_start and data_end bounds. Blank rows between groups are skipped, not treated as terminators.
  6. Mathematical identity checks. Pass 7 verifies Return of Capital + Preferred Return + GP Catch-up + Carry Base ≈ Total Proceeds. If the identity breaks, the audit log surfaces it loudly.
  7. Provenance for every number. Every DerivedMetric row carries formula_expression, gemini_reasoning, inputs_used (source cell + label + sheet), and confidence. The frontend renders these in a side panel on click — the user can audit any number.
  8. Pass 8 last-mile sanity check. For waterfall, even though Gemini computed gross / net / clawback, Python re-checks net == gross − clawback. If Gemini contradicts itself, Python backfills using the identity.
