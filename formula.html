1.  Preface & Scope

1.1  Purpose of this Document

This reference manual catalogues every quantitative formula used by the TrackFundAI dashboard to compute fund-performance, waterfall, capital and NAV metrics. Each metric entry provides the meaning, the exact computation formula as implemented in the Python aggregator, the source data ladder, and the priority order applied when multiple sources are available.

The formulas below are drawn directly from the production Phase 6 architecture and reflect the actual behaviour of the aggregator (phase4_derivations.py), the persister (phase2_persister.py), and the frontend metric-copy dictionary (v5-dashboard.js). They are the authoritative reference for auditors, analysts and reviewers of the platform.

1.2  Architecture Overview — Phase 6

The Phase 6 pipeline uses Google Gemini 2.5 Flash exclusively for classification — identifying which sub-sheet of an uploaded workbook corresponds to which fund-data domain and which column contains which canonical field. All numeric computation is performed deterministically in Python against the atomic ledger rows persisted to the database. Gemini is never used to compute, derive or fabricate numeric values.

The compute pipeline runs in four distinct stages:

  1. Stage 1 — Extraction: Gemini classifies sheets and maps columns; the unified builder emits a JSON extraction record.
  2. Stage 2 — Persistence: Atomic ledger rows are written to CapitalCall, Distribution, Investment, Valuation, ExitEvent, NAVRecord, Commitment.
  3. Stage 3 — Aggregation: Python computes all derived metrics using deterministic formulas.
  4. Stage 4 — Reconciliation: Computed values are cross-checked against workbook-extracted values; the higher-confidence source is preserved with a method tag.

1.3  Universal Rules of Computation

  • Computation-First: Every metric attempts a deterministic Python calculation from atomic ledger rows before falling back to an extracted cell value.
  • No Fabrication: Gemini proposes ranked candidate formulas only; Python evaluates them against catalogue values. Numeric outputs from the AI layer are never trusted directly.
  • Fund-Level Called Capital: Total Called Capital is always the sum of atomic CapitalCall.total_call_amount rows — the fund-level ledger figure. LP-cumulative totals from the register are used only as a Layer B fallback when the CapitalCall ledger is empty or materially inconsistent.
  • Temporal Consistency: All time-dependent formulas (Preferred Return, XIRR, Runway) share a single as_of date per fund, chosen by a priority ladder that rejects stale close-anchored dates.
  • Sanity Gates: Out-of-range values are rejected with logged reasons rather than silently clamped, so anomalies can be investigated.
  • Graceful Degradation: When inputs are insufficient the metric is emitted as blank with an itemised reason, never as a fabricated zero or a partial computation.

2.  Fund-Level Performance Metrics

2.1MOIC — Portfolio Multiple on Invested Capital Multiple (×)

How many times the fund has grown its invested capital across the entire portfolio.

Formula

Portfolio MOICMOIC = (Total Realised Proceeds + Residual NAV on Holding Basis) ÷ Total Invested Capital = (Σ ExitEvent.proceeds + Σ latest Valuation.fair_value_of_holding per Investment) ÷ Σ Investment.total_invested

Priority Ladder

P1 If all three inputs (realised proceeds, residual FV on holding basis, invested cost) are present, compute directly.
P2 If residual FV is unavailable, substitute the extracted fund_nav as fallback, subject to the sanity check that fund_nav does not exceed 5× invested capital.
Data Sources: ExitEvent.proceeds  |  Valuation.fair_value_of_holding (latest per Investment)  |  Investment.total_invested
2.2TVPI / Net MOIC — Total Value to Paid-In Multiple (×)

Total value created per rupee called from LPs (LP-perspective metric, ILPA standard).

Primary Formula

TVPI — Net MOICNet MOIC = (Total Distributions + Residual NAV) ÷ Total Called Capital

Priority Ladder (Four Tiers)

P1 Distributions present: Net MOIC = (Total Distributions + Residual NAV) ÷ Total Called Capital.
P2 Pre-distribution fallback: Net MOIC = Fund NAV ÷ Total Called Capital — used when distributions have not yet begun.
P3 Extracted cell: Value taken directly from a workbook "TVPI" or "Net MOIC" cell when present and passing the label semantic gate.
P4 Insufficient data: Metric blanked with an itemised list of missing inputs.
Data Sources: Distribution.total_net_amount (fallback total_gross_amount)  |  Computed Residual NAV  |  CapitalCall.total_call_amount  |  NAVRecord.total_nav (P2 only)
ILPA Identity: TVPI = DPI + RVPI when all three ratios share the same LP-perspective numerator basis. This identity is enforced by the persister and used as a runtime consistency check.
2.3DPI — Distributions to Paid-In Multiple (×)

How much capital has been returned to LPs per rupee called.

Formula

DPIDPI = Σ Capital Distributions ÷ Total Called Capital = Σ Distribution.total_net_amount ÷ Σ CapitalCall.total_call_amount

Layer B Fallback

When the atomic Distribution ledger is empty but the LP register carries cumulative distributed values, the fallback computation is:

DPI (fallback) = Σ Commitment.cumulative_distributed ÷ Total Called Capital
Data Sources: Distribution.total_net_amount (or total_gross_amount)  |  Commitment.cumulative_distributed (fallback)  |  CapitalCall.total_call_amount
2.4RVPI — Residual Value to Paid-In Multiple (×)

Unrealised value per rupee called — the still-held portfolio net of accrued GP carry.

Formula

RVPIRVPI = Residual NAV ÷ Total Called Capital

By construction, RVPI shares the same numerator basis as the residual-value component of TVPI, and the same denominator as DPI, so the ILPA identity TVPI = DPI + RVPI is preserved.

Data Sources: Computed Residual NAV  |  Total Called Capital (strict fund-level sum)
2.5Residual NAV ₹ Cr

The value LPs would actually receive if the fund liquidated today — the still-held portfolio at fair value, minus the GP's accrued carry.

Formula

Residual NAVResidual NAV = Σ latest Valuation.fair_value_of_holding per still-held Investment − Accrued Carried Interest Accrued Carried Interest = GP Carry Gross − Realised Carry Paid Out

Safeguards

  • Accrued carry is floored at zero; it never contributes a negative deduction.
  • When Σ fair_value_of_holding is zero or undefined, Residual NAV is emitted as blank with an itemised reason (no synthetic value).
  • Valuations flagged with methodology derived_from_cost_x_scheme_markup are excluded to prevent auto-filled mark-to-cost rows contaminating the fair-value sum.
Data Sources: Valuation.fair_value_of_holding  |  CarriedInterest.carry_amount_gross  |  CarriedInterest.gp_carry_distributed
2.6Net IRR — Internal Rate of Return Percent (%)

The annualised, time-weighted return that LPs earn after fees and carry.

Formula — Calculated (Probe A)

Net IRRNet IRR = XIRR(cashflows) where cashflows = Outflows: (CapitalCall.call_date, − CapitalCall.total_call_amount) Inflows: (Distribution.distribution_date, + Distribution.total_net_amount) Terminal: (as_of_date, + Terminal NAV) Terminal NAV = Σ latest Valuation.fair_value_of_holding per Investment (fallback: extracted Fund NAV, capped by sanity gate)

Formula — Extracted (Probe B)

B1 Gemini-classified fund_performance.net_irr_stated cell with provenance.
B2 Python workbook scan for the label "Net IRR" and its adjacent numeric value.

Four-Case Decision Matrix

Case 1 Both calculated and extracted present — prefer calculated; report both if they disagree by more than one percentage point.
Case 2 Only calculated present — use calculated.
Case 3 Only extracted present — use extracted (calculated inputs incomplete).
Case 4 Neither present — metric blanked with itemised reason.

Sanity Gates

  • IRR bounds: −99.99% to +999.99%; values outside this range are rejected. A physical cap at 500% is applied since no legitimate AIF IRR exceeds this level.
  • Capital-called sanity: called ≥ 50% of invested (a physics check).
  • Terminal NAV must be strictly positive.
Computation: XIRR by bisection over annual rate range [−0.99, +10.0], ~80 iterations, 1e−12 precision.
2.7Gross IRR — Per-Investment Percent (%)

Annualised IRR for a single portfolio investment, before fund-level fees and carry.

Formula

Gross IRR (per Investment)Gross IRR = XIRR over per-investment cashflows Outflows: (InvestmentTranche.date, − InvestmentTranche.amount) Inflows: (ExitEvent.exit_date, + preferring net_exit_proceeds, fallback proceeds) Terminal: (latest valuation date, + Valuation.fair_value_of_holding)

Universal Precedence Rule

If the extractor has already persisted a per-investment IRR from a workbook column labelled "IRR%(Gross)" or equivalent, that value is preserved and the XIRR recomputation is suppressed. This honours the fund manager's stated IRR when available.

Stored On: Investment.irr_pct  |  ExitEvent.irr_on_exit  |  Bounds: −99.99% to +999.99% (out-of-range clamped to None)

3.  Waterfall & Carried-Interest Metrics

3.1Carry Base ₹ Cr

The pool of profit above return-of-capital from which the GP's carry is calculated.

Formula — European Whole-Fund with 100% GP Catch-Up

Carry BaseCarry Base = max(0, (Realised Exit Proceeds + Residual NAV) − Total Called Capital)

Priority Ladder

P1 Extracted directly from a workbook cell (e.g. Carry_Clawback, Fund_Overview, WATERFALL_EUR).
P2 Computed by the universal formula above.
P3 Insufficient data — metric blanked with itemised missing inputs.
Data Sources: ExitEvent.proceeds  |  Valuation.fair_value_of_holding (residual)  |  CapitalCall.total_call_amount
3.2GP Carry Gross ₹ Cr

The gross performance fee earned by the GP for outperforming the hurdle.

Formula

GP Carry GrossGP Carry Gross = (carry_pct ÷ 100) × Carry Base = (carry_pct ÷ 100) × max(0, (Realised + Residual NAV) − Called)

Priority Ladder

P1 Extracted from a workbook cell — the CA-audited number takes precedence.
P2 Computed via the formula above (Carry Base × carry percentage).
P3 Insufficient data — metric blanked with itemised reason.
Data Sources: Scheme.carry_pct (LPA term)  |  Computed Carry Base
3.3GP Carry Net ₹ Cr

The GP's indicative take-home carry after the LPA-mandated clawback reserve is withheld.

Formula — AIF Standard

GP Carry NetNet Carry = Gross Carry × (1 − Clawback Reserve % ÷ 100)

Priority Ladder

P1 Extracted "Net Carry" or "Carry Payable" cell (CA-audited number).
P2 Computed via the AIF-standard formula above.
P3 Insufficient data — either Gross Carry missing or Clawback percentage unknown.
Data Sources: Computed GP Carry Gross  |  Scheme.gp_holdback_pct (LPA term)
3.4Preferred Return (Hurdle) ₹ Cr

The minimum annualised return — typically 8% compounded — that LPs must receive before the GP earns any carry.

Formula — Per-Call Accrual

Preferred ReturnPreferred Return = Σ CapitalCall.total_call_amount × ((1 + hurdle_rate)^years − 1) where years = (as_of_date − CapitalCall.call_date) ÷ 365.25
Data Sources: Scheme.hurdle_rate_pct (LPA term)  |  CapitalCall.call_date, CapitalCall.total_call_amount  |  as_of date (Solution E ladder)
3.5GP Catch-Up ₹ Cr

The 100%-to-GP tier that follows the LP preferred return, allowing the GP to catch up to its carry entitlement.

Formula

GP Catch-UpCatch-Up = min( Preferred Return × (carry_pct ÷ (1 − carry_pct)), max(0, Total Distributed − Return of Capital − Preferred Return) )

The catch-up tier is a step in the classic European waterfall and is only computed when both the hurdle percentage and the distribution ledger are present.

Data Sources: Computed Preferred Return  |  Scheme.carry_pct  |  Distribution ledger  |  CapitalCall ledger
3.6Clawback Provision ₹ Cr

Amount reserved from gross carry against potential repayment if the fund subsequently underperforms.

Formula

ClawbackClawback = max(0, GP Distributed − GP Carry Entitlement)

GP-distributed amounts are aggregated from two possible sources: Distribution.gp_carry_amount when the workbook splits the column, and Distribution.total_net_amount with distribution_type = 'carry' when carry is emitted as a separate ledger row.

Data Sources: Distribution.gp_carry_amount  |  Distribution rows with type='carry'  |  Computed GP Carry Gross
3.7Return of Capital ₹ Cr

The first tier of the European waterfall: LPs receive 100% of their called capital before any profit is split.

Formula

Return of Capital (Step 1)Return of Capital = min(Total Called Capital, Total Distributed)
3.8Management Fee Percent (%)

The annual fee charged by the GP for administering the fund.

The management fee percentage is a static LPA term persisted on Scheme.management_fee_pct. It is extracted from the workbook (typically from a MASTER_INPUTS, Fund_Overview or LPA-terms cell) and is not the subject of a computation.

Data Source: Scheme.management_fee_pct — extracted LPA term

4.  Capital Metrics

4.1Total Commitments (Committed Capital) ₹ Cr

The total amount that LPs have legally committed to invest in the fund.

Total CommitmentsTotal Commitments = Σ Commitment.commitment_amount
Data Source: Commitment.commitment_amount (per LP register row)
4.2Total Called Capital ₹ Cr

How much of committed capital has actually been drawn down and paid into the fund.

Primary Formula

Total Called CapitalTotal Called = Σ CapitalCall.total_call_amount

Layer B Fallback

When the atomic CapitalCall ledger is materially inconsistent with the LP register — specifically when the LP register's cumulative-called sum is more than twice the CapitalCall sum — the fallback is:

Total Called (fallback) = Σ Commitment.cumulative_called
Universal Rule: The dashboard always uses the fund-level atomic figure as its primary source. LP-cumulative totals are used only as fallback when the primary ledger is unavailable, and their use is logged with a diagnostic reason.
Data Sources: CapitalCall.total_call_amount  |  Commitment.cumulative_called (fallback)
4.3Uncalled Commitments ₹ Cr

The remaining capital LPs must contribute if the GP issues further calls.

Uncalled CommitmentsUncalled = max(0, Total Commitments − Total Called Capital)
4.4Total Invested Capital (Deployment) ₹ Cr

The total capital deployed into portfolio companies at cost basis.

Total Invested (Deployment)Total Invested = Σ Investment.total_invested
Data Source: Investment.total_invested (per portfolio-company investment row)
4.5Total Distributions ₹ Cr

The total capital returned to LPs, net of fees and carry.

Priority Ladder

P1 Atomic Distribution ledger: Total Distributions = Σ Distribution.total_net_amount (fallback to total_gross_amount when net is unavailable).
P2 Extracted aggregate: Value taken directly from a workbook cell such as Fund_Overview "Total Distributions".
P3 LP register rollup: Total Distributions = Σ Commitment.cumulative_distributed.
Data Sources: Distribution ledger  |  CarriedInterest.total_distributions  |  Commitment.cumulative_distributed
4.6Total Realised Proceeds ₹ Cr

The gross cash returned to the fund from all exit events.

Preference Order

P1 Total Realised = Σ ExitEvent.net_exit_proceeds.
P2 Fallback: Total Realised = Σ ExitEvent.proceeds (when net proceeds are unavailable).
Data Sources: ExitEvent.net_exit_proceeds  |  ExitEvent.proceeds (fallback)

5.  NAV & Valuation Metrics

5.1Fund NAV ₹ Cr

The audited net asset value of the fund at the reporting date.

Formula — Universal AIF NAV (P1, all components present)

Fund NAV — P1 ComputedFund NAV = Realised Gains on Exits + Unrealised Value (Residual portfolio at fair value) + Cash & Cash Equivalents + Receivables (Interest / Dividend + Other) − Management Fee Payable − Carry Payable − Other Liabilities (Fund Expenses + Tax + Borrowings)

Fallback Formula — Extracted (P2)

Fund NAV — P2 ExtractedFund NAV = extracted "TOTAL FUND NAV" cell from the workbook

Component Sources

ComponentSource
Realised GainsΣ ExitEvent.realized_gain_loss (fallback ExitEvent.proceeds)
Unrealised ValueΣ latest Valuation.fair_value_of_holding per Investment (excluding synthetic mark-to-cost)
Cash & Cash EquivalentsNAVRecord.cash_balance or Gemini-extracted cell
ReceivablesExtracted cells: Interest / Dividend receivable + Other receivables
Management Fee PayableExtracted workbook cell
Carry PayableExtracted or computed (per Section 3.2)
Other LiabilitiesExtracted cells: fund expenses + tax provision + borrowings
5.2NAV Per Unit ₹ Lakhs per unit

The per-unit net asset value, reported in Rupees Lakhs.

Priority Ladder

P1 Computed: NAV / Unit (₹ Lakhs) = Fund NAV (₹ Cr) × 100 ÷ Total Units Issued.
The factor of 100 converts Crores to Lakhs.
P2 Extracted: Value taken directly from a workbook per-unit cell, with unit-sanity normalisation — if the extracted value exceeds 10,000 it is treated as raw Rupees and divided by 100,000 to normalise to Lakhs.
P3 Insufficient data: Metric blanked.
Data Sources: Computed Fund NAV  |  Extracted "Total Units Issued" or NAVRecord.total_units_outstanding
5.3Total Fair Value ₹ Cr

The full value the fund has produced across its lifetime — still-held portfolio at fair value plus cash returned from exits.

Total Fair ValueTotal Fair Value = Active Fair Value + Realised Proceeds from Exits = (Σ Valuation.fair_value_of_holding per Investment) + (Σ ExitEvent.proceeds)
5.4Active Fair Value (Unrealised Portfolio) ₹ Cr

The fund's stake in every portfolio company it still owns, marked to fair value.

Two Bases of Measurement

Portfolio FV — Equity Value BasisPortfolio Fair Value = Σ Valuation.fair_value per Investment (matches the "Total FV Unrealised" column on an Excel Cover sheet)
Active FV — LP-Holding Basis (ILPA)Active Fair Value = Σ Valuation.fair_value_of_holding per Investment (falls back to fair_value when the holding column is missing) (matches what LPs would receive if the fund liquidated today)

Safeguard

Both aggregations exclude valuations flagged with methodology derived_from_cost_x_scheme_markup, which are auto-filled mark-to-cost rows that would otherwise contaminate the fair-value sum.

5.5Unrealised Value (AIF Section-A) ₹ Cr

Gross fair value of the still-held portfolio, as reported in AIF Section-A.

Unrealised ValueUnrealised Value = Σ latest Valuation.fair_value_of_holding per still-held Investment

AIF Section-A convention reports gross fair value, not mark-to-market gain. The cost basis is shown separately alongside for reference.

5.6Realised Value (AIF Section-A) ₹ Cr

Gross cash returned from all exits, as reported in AIF Section-A.

Realised ValueRealised Value = Σ ExitEvent.proceeds (gross cash before cost-basis deduction)

AIF Section-A convention reports gross proceeds, not the profit component. The profit portion is shown separately for reference.

6.  Fallback Patterns & Sanity Gates

6.1  The First-Present Pattern

Multi-source fallback chains are implemented via a helper that returns the first non-null value from an ordered list of candidates, or null if all candidates are null. This preserves zero values as valid answers and only rejects true absences:

_first_present(val1, val2, val3, ...) → the first non-null value, or null if all are null

6.2  Extraction Precedence

  1. Cell-verified overrides — Python re-reads the exact cell from the workbook cache to verify the extracted value.
  2. Trusted extractions — Gemini-emitted values that pass the label-whitelist and semantic gate.
  3. Python-computed formulas — Deterministic ledger-based calculation.
  4. Blank with reasons — An honest "—" plus an itemised missing-inputs table.

6.3  Sanity Bounds

MetricBoundsBehaviour on Violation
IRR−99.99% ≤ IRR ≤ 999.99% (physical cap 500%)Rejected to null with logged reason
MOIC0 ≤ MOIC ≤ 100×Rejected to null with logged reason
Fund NAVfund_nav ≤ 5× investedTreated as extraction error; used only as IRR-terminal fallback
Capital Calledcalled ≥ 50% of investedIRR computation aborted with reason (physics check)
NAV Per Unitvalue > 10,000 → treated as raw ₹Normalised to Lakhs by dividing by 100,000

6.4  Temporal Anchoring (Solution E)

All time-dependent formulas share a single as_of date resolved via the following priority ladder:

  1. Caller-supplied as_of parameter.
  2. Pre-existing CarriedInterest.calculation_date, provided it is more than six months after the fund close date (rejecting stale close-anchored dates).
  3. Latest NAVRecord.nav_date, provided it is more than six months after the fund close date.
  4. Today's date.

8.2  Provenance & Method Tags

Every major metric emits a method code alongside its numeric value, persisted to FundMetric.inputs_used. The frontend uses these tags to transparently label which tier of the priority ladder was used. Representative tags include:

  • p1_extracted_cell — value taken from a verified workbook cell
  • p2_computed_formula — value computed by the standard Python formula
  • p3_insufficient_data — value blanked; reason recorded
  • p1_computed_with_distributions — TVPI computed using the distribution ledger
  • p2_computed_fund_nav_over_called — TVPI computed via Fund NAV fallback
  • lp_register_cumulative_distributed — distributions rolled up from the LP register

8.3  Perspective Distinction

Metrics can be reported on two distinct bases:

  • Portfolio (Equity Value) Basis — uses Valuation.fair_value, matching the Excel Cover sheet's total unrealised column. This is the basis for Portfolio MOIC.
  • LP (Holding) Basis — uses Valuation.fair_value_of_holding, matching the ILPA convention of the fund's own stake. This is the basis for TVPI, DPI, RVPI and Net IRR.

8.4  Waterfall Model

The dashboard implements the universal European Whole-Fund waterfall with 100% GP catch-up. The tiers are, in order: (i) return of called capital to LPs, (ii) preferred return to LPs at the LPA hurdle rate, (iii) 100% catch-up to the GP until it reaches its carry entitlement, and (iv) profit split at the carry percentage. Where LPA terms diverge from this model, the extracted cell values take precedence over the computed waterfall.
