TrackFundAI — Anchor-First Deterministic Pipeline

Production architecture replacing the 9-pass Gemini chain. Last updated: 2026-06-10.

1 · The problem this architecture solves

The previous pipeline ran nine sequential AI passes (Pass 1 → Pass 1.5 → Pass 2 → Pass 2.5 → Pass 2.6 → Pass 3 → Pass 3.5 → Pass 4 → Pass 8 → Pass 9) plus a MetricArbiter. On structured fund files it produced wrong values by ₹100s of crores because (a) each pass introduced its own classification errors, (b) the Arbiter could only pick the "least wrong" candidate, and (c) the pipeline asked Gemini to do fuzzy semantic matching when the right task was arithmetic from a few anchor values.

The replacement runs in four stages, makes ≈3 Gemini calls per scheme, and is fully deterministic: same file in → same numbers out, every time.

Before

  • 9 passes + Arbiter
  • ~30+ Gemini calls per import
  • Catalogue formulas hardcoded
  • Domain-classification gatekept everything
  • Stochastic re-imports — values varied
  • ₹100s of Cr wrong

After

  • 4 stages, anchor-first
  • 3-4 Gemini calls per import
  • Textbook formulas only (no catalogue)
  • No domain gatekeeping — whole workbook fed to Gemini
  • Deterministic re-imports
  • Audited accounting identities

Design rule

  • Zero hardcoded value ranges
  • Zero keyword lists
  • Zero sheet-name assumptions
  • Gemini reasons by financial role, not labels
  • Python does math only
  • Every output has provenance

5 · How the architecture handles unlabelled, unordered, chaotic files

The architecture makes no assumptions about file structure. Everything that determines a metric value comes from one of three places:

  1. Gemini's reasoning about financial role (Stages 1-3)
  2. Python's textbook formula (Stage 5)
  3. Python's accounting identity (Stage 6)

None of these depend on sheet names, column names, row positions, or language. Below is the resilience contract:

Adversarial file propertyWhy the pipeline still works
Sheets named in another language ("バランスシート" vs "Balance Sheet") Stage 0 census reads every cell regardless of sheet name. No domain classification depends on sheet name in this pipeline.
Columns reordered (Cost might be col B in one file, col F in another) Gemini Stage 2 reasons about the value's role, not its column position. It uses adjacency, magnitude, and cross-sheet consistency to identify each anchor.
Labels missing entirely (just numbers, no row headers) Stage 2 prompt asks Gemini to use accounting relationships (e.g., "Committed ≥ Called", "LP commitments sum to Committed") as evidence. The values triangulate even without labels.
Labels misleading (a column labeled "MOIC" actually contains TVPI) Stage 5 computes MOIC from anchors, ignoring whatever the file calls "MOIC". If the file states a value too, Stage 6 cross-checks and flags conflict.
Currency in millions but file says "Cr" somewhere Stage 1 returns ALL currency/unit candidates with cell locations. Stage 5 normalises consistently.
Multiple plausible candidates for an anchor Stage 2 returns all candidates ranked. resolve_anchors() in Python picks the one consistent with accounting identities (Committed ≥ Called ≥ Invested).
Sheets reordered (NAV might be sheet 1 in one file, sheet 12 in another) Census serialises all sheets into one JSON; Gemini sees the full workbook context in every call.
Sub-sheets with different layering (per-LP vs fund-summary) Stage 3 cash-flow prompt explicitly says: "If both per-LP and fund-level summary exist, use fund-level to avoid double-counting." Stage 2 same principle.
No labels for hurdle / carry % (only used inside formulas) Stage 2 asks Gemini to infer from formula context — e.g., if a column computes Capital × ((1+0.08)^years − 1), that's hurdle = 8%.

6 · Semantic analysis at each level

Sheet-level semantics

The anchor pipeline does not classify sheets. Every sheet's contents are flattened into the census payload and Gemini sees the full workbook. This is intentional — the previous pipeline's domain classification was the gatekeeper that failed for Multiples WATERFALL.

Per-row CRUD lane does still use Pass 1 sheet → domain mapping in gemini_column_mapper.py, but only for portfolio-companies / investors / KPIs / NAV time-series.

Section-level semantics

Stage 2 prompt names what each anchor is ("the cumulative capital actually drawn from investors") and Gemini locates it from the cell context — surrounding labels, sub-totals, row position relative to other anchors. No keyword list; the model uses semantic understanding of the financial concept.

Row-level semantics

For each anchor, Gemini returns up to 3 ranked candidates with the row's surrounding text as evidence. Python's resolve_anchors() picks the candidate consistent with locked-in earlier anchors via accounting identities.

Column-level semantics

For multi-period sheets (NAV history, BvA, KPIs), the per-row CRUD lane uses Pass 2 column mapping. The fund-level lane (this pipeline) does not need column mapping — Gemini sees raw cell values in their addresses and reasons directly.

Field-level semantics

Every fund metric maps to a single canonical key (one of 22). Every anchor maps to one of 12 canonical names. No aliases, no synonyms, no fuzzy matching — the canonical schema is closed.

Conflict semantics

When a file states a value (e.g., Net IRR = 21.2%) that disagrees with the computed value (e.g., XIRR over extracted cash flows = 13.5%), both are kept in FundMetric.provenance and FundMetric.source is set to conflict. The dashboard tile shows the computed value with an audit icon.

9 · How we get to 100% accuracy

Failure mode (old pipeline)How the new pipeline prevents it
Pass 1 mis-classifies a sheet (e.g., WATERFALL not tagged as waterfall_carry), Pass 8 sees nothing → produces zero candidates → Arbiter falls to garbage Pass 4 No sheet classification gate. Stage 0 census includes every sheet. Stage 2 prompt sees the whole workbook and identifies carry-related values regardless of sheet name.
Pass 4 catalogue formula picks a random cell because the label is fuzzy No catalogue formulas. All metric computation is textbook math on validated anchors. If anchor is null, metric is null (honest) — never a guess.
Variant key collision: both Portfolio FV and Fund NAV stored as total_unrealised_fair_value Closed canonical schema with 12 distinct anchor names. active_fair_value ≠ fund_nav; the prompt explicitly disambiguates.
Re-import produces different values (non-determinism) Gemini temperature=0, JSON mode. Same prompt + same census → same output. Stage 5-7 are pure Python. Persist is atomic with delete-then-insert per scheme.
Cross-Pass races + last-writer-wins on DerivedMetric Single writer: persist(). No competing writers.
Silent failures (Pass returns empty, no log) Every stage logs at INFO level. Audit findings are recorded in DerivedMetric.gemini_reasoning. Stage 2 conflicts (no candidate satisfies identities) are logged WARN.
File states MOIC = 2.0 but math says 0.95 — system picks the stated one Computed value always wins. If the file's stated value disagrees beyond 1%, FundMetric.source = conflict and the audit icon appears on the tile. CFO reviews — system never silently picks the wrong one.
Currency unit mixed (some sheets in Cr, some in Lakhs) Stage 1 returns all unit candidates with cell locations. compute_metrics uses values raw; any inconsistency surfaces in audit identities (Called ≤ Committed fails if units are mixed).
