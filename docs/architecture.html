1. Executive Summary

TrackFundAI ingests Indian AIF Excel workbooks of arbitrary GP/format/vintage and produces a multi-tenant fund management dashboard. The platform has gone through three pipeline generations:

GenerationApproachStatusTradeoffs
Phase 1
Deleting		 Multi-pass Gemini chain: classify sheets → classify sections → map columns → interpret values → derive metrics. ~9,400 lines in import_service.py. Disabled — being permanently removed Slow (6+ Gemini calls), fragile chain, hardcoded keywords for 20+ categories.
Single-call extractor
Current		 One Gemini Pro call ingests the entire workbook and returns a structured JSON. Production default since 2026-05; gated by USE_NEW_EXTRACTOR=true Hits 65K output-token ceiling on large workbooks. No semantic layering — one prompt does everything.
Phase 3 (Flavor A + B)
Design		 Always 3 parallel Gemini calls (Layer 1 Identity + Layer 2 Universe + Layer 3 Time-series). Flavor B fires inside any layer that overflows — either pre-emptively (estimator predicts > 35K output tokens) or reactively (Gemini's response is detected as truncated mid-call, then the chunk auto-splits into two and reruns). Design approved; implementation gated on priority_matrix.md sign-off Preserves single-call latency (wall time = MAX of parallel calls), bounded by output token ceiling, scales to 500+ company portfolios.
Why semantic layering, not row chunking as default? Row chunks break formula consistency (e.g., per-LP IRR rows need fund-level NAV context). Semantic layering keeps related fields in the same call so cross-references stay coherent. Row chunking is reserved for genuinely huge time-series sheets where rows are independent observations.

5. Flavor A — Semantic Layering

Flavor A always runs 3 parallel Gemini calls. Each call is given the same identity context (Cover sheet + Fund Master extracted by a one-shot pre-pass, or replicated as a copy of the L1 prompt preamble) so it has independent context — no layer depends on another layer's output.

LAYER 1
Identity & Fund-Level

Cover, Fund Master, Investors, Capital Calls, Distributions, NAV Walk, Waterfall, Fund Performance. Emits scalar identity + fund-level metrics with provenance.

LAYER 2
Investment Universe

Portfolio Companies, Investments (per-scheme), Tranches, Valuations, Exits, Quoted/Unquoted. Emits per-investment data with tranche-level provenance.

LAYER 3
Time-Series

Per-company monthly KPIs (revenue, EBITDA, GMV, MRR, etc.), monthly P&L / BS / CF, burn & runway. Highest output-token risk → Flavor B activates here first.

Layer 1 — Identity (deep dive)

Sheets routed here
Any sheet whose router-classification ∈ {cover, fund_master, investors, capital_calls, distributions, nav_walk, waterfall, fund_performance, scheme_lifecycle, entities}.
Prompt structure
Common rules (Rules 1–34) + Layer 1 instructions + identity context replication.
Expected output blocks
fund_master, scheme_terms, entities[], investors[], commitments[], capital_calls[], distributions[], nav_records[], fund_performance{...}.
Cross-layer dependencies
None outbound (L1 is foundational). Cross-layer reconciliation may compare L1's fund-level aggregates against L2's row-sums (P3 principle).

Layer 2 — Universe (deep dive)

Sheets routed here
Any sheet whose classification ∈ {portfolio, valuations, tranches, exits, quoted_unquoted}.
Output blocks
portfolio_investments[], valuations[], tranches[], exits[].
Flavor B trigger
Pre-emptive: token estimator (~8 tokens per source cell + 3K overhead) > 35K → pre-chunk into N parallel sub-calls.
Reactive: if Gemini truncates a chunk mid-response (finish_reason = MAX_TOKENS), _run_chunk auto-splits that chunk in two and reruns each — recursively up to 4 levels. Either path guarantees Layer 2 data lands.

Layer 3 — Time-series (deep dive)

Sheets routed here
Any sheet whose classification ∈ {kpi_matrix, monthly_pl, monthly_bs, monthly_cf, budget_vs_actual, company_financials, burn_runway}.
Output blocks
portfolio_kpis_periodic[], monthly_pl[], monthly_bs[], monthly_cf[].
Flavor B trigger
Highest output-volume layer. Pre-emptive split fires when est ≈ (# companies × # periods × # line-items × 8 tokens) > 35K. Even when pre-split fires correctly, any chunk that still truncates auto-splits again — the recursive safety net is identical to L2's.
Chunk dimension
Row-range chunking by company group (preferred — keeps a company's time series intact) OR by period range (only when one company has >100 periods).

Dynamic Layer Formation — How Sheets Get Routed

The router (workbook_router.py) classifies every sheet into L1/L2/L3 using a lightweight rules-only classifier (no Gemini — runs in milliseconds). The classification cascade:

  1. Header keyword scan — first 5 rows of each sheet scanned against domain-specific anchor terms (e.g., "NAV", "Carried Interest", "Capital Account" → L1; "Portfolio", "Investment", "Valuation" → L2; "Monthly", "KPI Matrix", "P&L", "Burn" → L3).
  2. Column-count + row-count heuristic — wide-many-row sheets (cols ≥ 8, rows ≥ 30) → L2 or L3; narrow tall (cols ≤ 4) → L1.
  3. Period-detection — if header row contains ≥ 3 date-pattern cells → L3 (time-series).
  4. SHEET_DOMAINS lookup — existing canonical_schema.SHEET_DOMAINS (kept from Phase 1) gives high-confidence overrides.
  5. Fallback — anything unclassified is added to L1 (safest layer; small marginal token cost).
No Gemini call for routing. Classification is deterministic, fast, and fully auditable. If the router misclassifies a sheet, the layer prompt will still ignore it (because the prompt asks only for fields that belong in that layer), so the worst case is one wasted parse pass — never wrong data.

6. Flavor B — Row Chunking (Detect-and-React)

The honest truth about predicting Gemini's output size. A pre-call token estimator can never be 100% accurate. The output depends on what Gemini actually decides to emit — how verbose its provenance text is, which optional fields it includes, how the prompt's instructions are interpreted. We can't know the output size until we have the response. Anyone claiming "100% accurate prediction" is lying. Instead, Flavor B guarantees 100% safe extraction via a detect-and-react architecture where prediction is just the first guess and three independent safety nets catch every overflow.

Flavor B — in 5 simple steps

  1. Estimate first, just as a head start. The token estimator looks at the size of the source sheets routed to a layer and predicts how big the Gemini output will be. If the prediction is above 35,000 tokens, Flavor B pre-splits the layer into smaller chunks before making any call — this saves wall time when we already know one big call would fail.
  2. Make the Gemini call. Each chunk is sent to Gemini as a normal parallel call inside the shared ThreadPoolExecutor. We do not try to second-guess Gemini's behaviour at this point — we just let it run.
  3. Detect truncation with three independent signals. After the call returns, we check whether the response was cut off:
    • Signal 1 — Gemini's own flag. The Vertex AI SDK sets response.candidates[0].finish_reason = MAX_TOKENS when Gemini hit its output ceiling. This is the gold-standard signal: if Gemini says "I was truncated", we believe it 100%.
    • Signal 2 — JSON parse failure mid-array. If the response ends abruptly (no closing brace) or has malformed JSON, that's circumstantial evidence of truncation.
    • Signal 3 — Token usage ratio. If output_tokens is at ≥ 95% of the model's maximum, the response is suspiciously close to the limit — likely truncated even if it happens to parse.
  4. If truncated, split the chunk in two and rerun automatically. When any signal fires, the orchestrator does not give up. It splits the chunk's row set into two non-overlapping halves and runs each as a new Gemini call. Splitting is mathematically clean: a chunk filtering row % N == K becomes two children row % 2N == K and row % 2N == (K+N), so every original row is covered by exactly one child — no duplicates, no gaps. Each child can split again if it too truncates.
  5. Bottom out when a chunk is small enough that it MUST fit. Recursion is capped at PHASE3_MAX_SPLIT_DEPTH levels (default 4). A chunk that splits four times becomes 1/16th of the original row set — well below any plausible Gemini output ceiling. If a chunk somehow still truncates at this depth, only that tiny fragment is marked failed; the rest of the layer's data is preserved.
The key insight. The pre-call estimator can be off by 10× and the import will still produce 100% correct data, because the post-call detection-and-split layer catches every truncation. The estimator only affects speed (fewer wasted calls), not correctness.

Worked example — Mock_14_Multiples_IV.xlsx (the import that triggered this rewrite)

This is from a real failure on 2026-06-23 that drove the redesign. The workbook has 125 portfolio companies. Before the redesign, Layer 2 ran as one giant call, hit Gemini's 65K output ceiling, truncated mid-array, the entire L2 payload was lost, and the dashboard showed zero companies. With the new flow:

StageWhat happensCalls
EstimatorPredicts L2 output ≈ 40K tokens → pre-splits into 2 chunks (row % 2 == 0 and row % 2 == 1)2 parallel L2 calls
First chunk runsGemini emits clean JSON in ~25K tokens. finish_reason = STOP → no truncation → done.—
Second chunk runs in parallelGemini emits ~32K tokens but is unexpectedly verbose; finish_reason = MAX_TOKENS.—
DetectionSignal 1 fires (finish_reason). Orchestrator auto-splits into row % 4 == 1 and row % 4 == 3 at depth = 2.+2 sub-calls (sequential)
Both sub-calls succeed~16K tokens each, both finish_reason = STOP. Merger combines into the original L2.split_b slot.—

Why no overlap when splitting? The modulo split row%N==K → row%2N==K ∪ row%2N==(K+N) is provably disjoint and complete. Mathematically, every original row maps to exactly one child chunk — guaranteed by the structure of modular arithmetic, not by any manual list of row indices. The merger therefore concatenates child results losslessly.

7. Parallel Execution Model

All Gemini calls (Flavor A layers + Flavor B chunks) submit to one shared ThreadPoolExecutor. Default pool size = 8 workers (matches our Vertex quota burst budget). Calls return JSON futures that the orchestrator gathers with concurrent.futures.as_completed.

8. Reconciliation & Priority Matrix

The reconciler (reconciler.py) consumes the merged JSON and, for every field that exists in priority_matrix.md, picks ONE authoritative value using the matrix rules. The picked value goes into the unified output; alternative values are recorded in FundMetric.provenance.alternatives[].

Reconciler algorithm (per field)

  1. Load priority_matrix entry for the field.
  2. For each priority source (P1, P2, P3...), check if a value exists in the merged JSON.
  3. Pick the highest-priority source that has a value. Record priority_rule_applied = the principle(s) that determined this choice.
  4. For all lower-priority sources that also have a value: compare against the picked value.
    • Within tolerance → record as alternatives[] (informational, no flag).
    • Outside tolerance → record as disagreements[] and set quality_flag = 'reconciled' (⚠️ in side panel).
  5. Write the final value + provenance into FundMetric.
Single value, transparent provenance. Downstream calculations and dashboard rendering always see ONE authoritative value per field. Disagreements are visible (⚠️ icon + side-panel detail) but never block downstream math.

9. Cross-Layer Validator

Runs after reconciliation. Checks the mathematical identities listed in priority_matrix.md §Q:

  • total_committed_capital ≡ Σ Commitment.commitment_amount (±1%)
  • active_fair_value ≡ Σ latest Valuation.fair_value_of_holding (active) (±1%)
  • tvpi ≡ (distributions + nav) ÷ called (±0.05x)
  • rvpi + dpi ≡ tvpi (±0.05x)
  • nav_per_unit × total_units ≡ total_nav (±0.5%)
  • … and ~10 more identity checks

Violations are logged at WARN, written to FundMetric.provenance.identity_violations[], and surface as ⚠️ on the dashboard side panel. They do NOT block persistence — the import succeeds; the dashboard shows the values with a quality flag.

11. Frontend Render & Provenance

The dashboard never re-derives metrics. Every visible value reads from FundMetric via the fmValue('metric_key') helper. When a user clicks a tile, openProvenancePanel(metricKey) opens a side panel showing the value, the priority rule applied, the formula/extraction provenance, and any reconciliation disagreements.

Render chain

  1. Page load → loadOverview() calls /api/dataimport/derived-metrics/?scheme=<id>
  2. API returns FundMetric rows for the scheme, indexed by metric_key into _derivedMetrics
  3. fmValue('moic') returns _derivedMetrics['moic'].value (or null if absent)
  4. Tile render: el.textContent = fmCr('active_fair_value')
  5. wireProvenance('kv-moic', 'moic', displayed) binds click handler
  6. On click → openProvenancePanel('moic') renders panel from _METRIC_COPY['moic'] + FundMetric.inputs_used + FundMetric.provenance
