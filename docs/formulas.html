MOIC — Multiple on Invested Capital

Three MOIC formulas coexist because MOIC is asked in three different scopes (per-investment, fund-level, per-company aggregate). Each has its own inputs and rounding.

Fund-Level Portfolio MOIC Portfolio-basis multiple shown on the Cover tile
MOIC = (total_distributed + portfolio_residual) / total_invested
Where portfolio_residual = SUM(Valuation.fair_value) per active investment, with fund_nav as sanity-bounded fallback (rejected if >5× invested).
Quantized to 4 decimals. backend/dataimport/phase4_derivations.py:1181-1187
Per-Investment MOIC Written to Investment.moic during Phase 4 derivation
MOIC = SUM(positive cashflows) / SUM(negative cashflows)
Where positive = exit_proceeds + distributions + terminal_FV (if active), negative = tranche investments. Quantized to 4 decimals.
backend/dataimport/phase4_derivations.py:255-260
Per-Company Aggregate MOIC Computed on-demand by the DRF serializer for the Companies grid
MOIC = total_fv / total_cost
Where total_fv = SUM(exit proceeds) when exited, else SUM(inv.moic × inv.total_invested) for active investments; total_cost = SUM(Investment.total_invested).
backend/investments/serializers.py:125-130

Net IRR % — Priority Ladder (3 tiers)

Fund workbooks vary widely in what data they publish. Python attempts three IRR computations in priority order; the first that produces a plausible value wins. A 500 %/yr sanity cap discards absurd XIRR outputs at every tier. The chosen method is emitted via net_irr_method so the dashboard tile can label the source.

Tier 1 Preferred ILPA Net IRR — XIRR over LP-perspective ledger
IRR = XIRR([−CapitalCall, +Distribution, +Terminal_FV])
Requires: total_called ≥ 50 % of total_invested (physics bound — you cannot invest more than you have called).
Gives: industry-standard LP-perspective Net IRR.
backend/dataimport/phase4_derivations.py:1253-1330 (Tier 1 selector)
Tier 2 Fallback Cost-weighted per-investment Gross IRR
IRR = SUM(cost × irr_pct) / SUM(cost)
Requires: majority of investments have workbook-provided irr_pct.
Gives: portfolio-level Gross IRR (before fees/carry) — standard fund-manager reporting format.
backend/dataimport/phase4_derivations.py:1330-1385
Tier 3 Last resort Fund-level XIRR on Investment cashflows → LP terminal
IRR = XIRR([−Investment.cost, +Distribution, +Terminal_FV])
Requires: at least Investment.investment_date + total_invested exist.
Gives: approximation of Gross IRR from atomic ledger only. Used when call extraction is incomplete AND workbook doesn't publish per-inv IRR.
backend/dataimport/phase4_derivations.py:1385-1409
XIRR algorithm (common to all three tiers): pure-Python bisection over the annual rate. Bracket [−0.99, +10.0], widened geometrically (×4) up to 1e6 for extreme cashflows. 80 iterations → 1e-9 precision, ~1 ms per call. No numpy/scipy dependency. Returns Decimal percent (e.g. 27.5 for 27.5 %).
backend/dataimport/phase4_derivations.py:32-102 (_xirr)

TVPI — Total Value to Paid-In

DETERMINISTIC
TVPI = (total_distributed + residual_value) / perf_denom
residual_value = SUM(Valuation.fair_value_of_holding) per active investment, with fund_nav as sanity-bounded fallback.
perf_denom = total_called if total_called ≥ total_invested else total_invested (graceful degradation when call rows missing).
Never left blank if any capital base + any value present. Quantized to 4 decimals.
backend/dataimport/phase4_derivations.py:1217-1226

DPI — Distributed to Paid-In

DETERMINISTIC (STRICT ILPA)
DPI = total_distributed / total_called
Strict ILPA definition — denominator is called capital, not invested. Only computed when total_called > 0. Quantized to 4 decimals.
backend/dataimport/phase4_derivations.py:1207-1209

RVPI — Residual Value to Paid-In

DETERMINISTIC
RVPI = residual_value / perf_denom
Same residual_value and perf_denom definitions as TVPI (see above).
backend/dataimport/phase4_derivations.py:1211-1215

European Waterfall — RoC / Preferred Return / Catch-Up / Carry

Deterministic Python computation using extracted LPA terms (hurdle_rate_pct, carry_pct) + CapitalCall + Distribution ledgers. "Extracted-first, computed as fallback" — if the workbook's Carry_Clawback sheet published a verified cell value, that wins over the formula.

Step 1 — Return of Capital
step1_roc = MIN(total_called, total_distributed)
backend/dataimport/phase4_derivations.py:417
Step 2 — Preferred Return (compounded, per-call accrual)
preferred_return = SUM_over_calls[ call_amount × ((1 + hurdle_frac)^years − 1) ]
years = (as_of − call_date).days / 365.25, hurdle_frac = hurdle_pct / 100. Industry-standard compounded pref return.
backend/dataimport/phase4_derivations.py:424-435
Step 3 — GP Catch-Up (closed form)
catchup = preferred_return × carry_frac / (1 − carry_frac)
Equivalent to "100 % to GP until GP receives carry_frac of (pref + catchup)". Capped at remaining distributable after RoC + preferred return.
backend/dataimport/phase4_derivations.py:443-454
Carry Base & Gross Carry
carry_base = MAX(0, total_distributed − total_called)
carry_amount_gross = carry_frac × carry_base
backend/dataimport/phase4_derivations.py:439, 457
Step 4 — 80:20 Residual Split
remainder = MAX(0, distributable_after_roc_pref − catchup)
step4_lp = remainder × (1 − carry_frac)
step4_gp = remainder × carry_frac
backend/dataimport/phase4_derivations.py:460-462
GP Distributed, Clawback, Holdback, Net Carry
gp_distributed = catchup + step4_gp
gp_clawback = MAX(0, gp_distributed − carry_amount_gross)
gp_holdback = 0.20 × gp_distributed
carry_amount_net = carry_amount_gross − gp_clawback
Holdback default = 20 % of gp_distributed (SEBI/ILPA industry default; overridable per Scheme.LPA when field exists).
backend/dataimport/phase4_derivations.py:464-475

CAC — Customer Acquisition Cost (₹)

DETERMINISTIC
CAC = (Sales_and_Marketing_spend × 10,000,000) / new_customers
Multiplied by 1e7 because financials arrive in Crore (1 Cr = ₹1,00,00,000) but CAC is shown in raw INR (₹ per customer). Only computed when both S&M spend and new_customers > 0.
backend/dataimport/phase2_persister.py:3076-3078

LTV — Customer Lifetime Value (₹) & LTV/CAC

LTV (SaaS textbook formula)
LTV = (ARPU_cr × GrossMargin_frac / Churn_rate) × 10,000,000
ARPU_cr = ARR / customers (in Cr), GrossMargin_frac = (Revenue − COGS) / Revenue. Final ×1e7 converts Cr → INR.
Requires all of ARR, customers, churn_rate, Revenue, COGS to be non-null and > 0.
backend/dataimport/phase2_persister.py:3082-3088
LTV/CAC Ratio
ltv_cac_ratio = LTV / CAC
Universally computed the moment both LTV and CAC are populated.
backend/dataimport/phase2_persister.py:3091-3094

Terminal / Residual FV — Fair-Value Estimation Chain (universal helper)

Used by TVPI, RVPI, MOIC, Net IRR whenever a per-investment terminal value is needed. 3-step fallback chain — first hit wins.

Step 1 — Truth
FV = latest Valuation.fair_value_of_holding
Direct read of per-investment valuation row. Best signal when present.
Step 2 — Post-money × Ownership × Markup
FV = latest_tranche.post_money_valuation × ownership_fraction × fund_markup
ownership_fraction = fully_diluted_pct / 100 if > 1 (percent-vs-fraction detection).
Step 3 — Cost × Fund Markup
FV = Investment.total_invested × fund_markup
Where fund_markup = latest_NAV.investments_at_fair_value / SUM(Investment.total_invested). Falls back to 1.0 when NAV/cost signals missing.
backend/dataimport/phase4_derivations.py:106-177
Universality principle observed throughout: AI never generates numbers. Gemini classifies sheets and proposes candidate formulas; Python evaluates them using values already persisted from the workbook. Every formula above operates on Decimal types (no float drift) and quantizes to 4 decimals for storage. No fund-specific or sheet-specific logic — all formulas are format-agnostic and apply to any Indian AIF workbook.
